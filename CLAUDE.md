# PDR-I QA Toolkit

Risk-driven QA and defect prediction system for the PowerDirector iOS app (`com.cyberlink.powerdirector`). It fetches bug data, scores modules by risk, clusters bugs by theme, predicts defects, auto-generates test skeletons, and visualizes everything via a Streamlit dashboard.

## Tests

All files in `tests/` are placeholder skeletons generated by `auto_tag_tests.py` (stage 4 of the data pipeline). They define the class/method structure and Allure/pytest metadata but contain only `pass` stubs ‚Äî none are implemented. Do not attempt to run or extend these tests as functional code.

## Important: Testing and Verification

**Do not run the full pipeline to verify script changes.** A full run can take many minutes to complete. Instead:
- Use a small subset of real data (e.g., slice the first few rows of a CSV)
- Create dummy/synthetic data that exercises the specific code path being tested
- Use `--dry-run` to validate pipeline logic without executing stages
- Test individual scripts directly with minimal input rather than via `refresh_pipeline.sh`

## Pipeline Commands

```bash
# Run full data refresh pipeline
./refresh_pipeline.sh

# Faster partial runs
./refresh_pipeline.sh --skip-ollama                   # Use heuristic scorer instead of Ollama
./refresh_pipeline.sh --skip-cluster                  # Skip NLP clustering (~5 min savings)
./refresh_pipeline.sh --skip-predict                  # Skip ML defect predictions
./refresh_pipeline.sh --dry-run                       # Print steps without executing

# Schedule overrides
./refresh_pipeline.sh --weekday                       # Force weekday schedule (pdri + phdi, 1 month)
./refresh_pipeline.sh --weekend                       # Force full schedule (all products, 36 months)

# Product/duration overrides
./refresh_pipeline.sh --products pdri,phdi            # Specific products (comma-separated)
./refresh_pipeline.sh --products pdri --duration-months 3

# Model overrides
./refresh_pipeline.sh --ollama-model gemma4                 # Default LLM
./refresh_pipeline.sh --embed-model mxbai-embed-large       # Alternate embedding model
```

### Product Scheduling

The pipeline has a built-in schedule based on day of week:
- **Weekdays (Mon‚ÄìFri):** `pdri` + `phdi`, 1-month data window
- **Saturday only:** All products (`pdri`, `phdi`, `pdra`, `phda`, `pdr`, `phd`, `promeo`, `vvg`), 36-month window
- Sunday: skipped

Use `--weekday` / `--weekend` / `--products` to override.

## Architecture

### Data Pipeline (`scripts/`)

Scripts run in this fixed order per product ‚Äî each stage outputs CSV/JSON files consumed by the next:

1. `fetch_from_n8n.py` ‚Äî fetch raw bugs from n8n webhook **and** parse them inline (`--then-parse`); outputs `ecl_raw.json` + `ecl_parsed.csv`
2. `compute_risk_scores.py` ‚Äî compute risk metrics ‚Üí `risk_register_all.csv`
3. `ai_risk_scorer.py` ‚Äî AI scoring via heuristic / Ollama / Claude API ‚Üí `risk_register_scored_all.csv`
4. `auto_tag_tests.py` ‚Äî generate test skeletons in `tests/generated/<slug>/` + quadrant summary
5. `cluster_bugs.py` ‚Äî NLP clustering ‚Üí `ecl_parsed_clustered.csv` (skippable with `--skip-cluster`)
6. `predict_defects.py` ‚Äî ML forecasting ‚Üí `ecl_parsed_predictions.csv` (skippable with `--skip-predict`)

`parse_ecl_export.py` exists as a standalone utility for manually parsing Excel/JSON exports outside the pipeline.

`refresh_pipeline.sh` orchestrates all stages with atomic file promotion: writes to `data/staging/products/<slug>/`, validates, then `mv` to `data/products/<slug>/`. Each product is isolated ‚Äî a failure in one does not stop others.

`bug_heatmap_dashboard.py` is the Streamlit UI (9 tabs) reading all pipeline outputs. It runs as a persistent process on the Mac Mini at `http://<hostname>:8501`.

`visual_regression.py` ‚Äî standalone visual diff/screenshot comparison tool; not part of the main pipeline.

`pages/02_Pending_Module_Review.py` ‚Äî Streamlit sub-page for reviewing fuzzy-matched module mappings (`*_pending.json`).

## Key Conventions

### Data File Naming

| Pattern | Stage |
|---|---|
| `ecl_*.csv` | Parsed bug data |
| `risk_register_*.csv` | Risk scoring output |
| `*_clustered.csv` | Bug clustering output |
| `*_predictions.csv` | Defect forecasts |
| `*_pending.json` | Unconfirmed module mappings (fuzzy-match candidates) |
| `quadrant_summary.md` | P1‚ÄìP4 module prioritization table |

### Appium Driver Config

Configured via environment variables (with defaults):

| Variable | Default |
|---|---|
| `APPIUM_URL` | `http://127.0.0.1:4723` |
| `BUNDLE_ID` | `com.cyberlink.powerdirector` |
| `DEVICE_NAME` | `iPhone` |

Driver has 10-second implicit wait and `no_reset=True` (preserves app state between tests). Session-scoped ‚Äî one driver instance for the entire test run.

### CI/CD (Jenkins)

The `Jenkinsfile` runs on `mac-mini-m1` agent. `TEST_SCOPE` parameter controls which quadrant stages execute. All stages use `|| true` ‚Äî failures don't block subsequent stages. Allure results are published from `allure-results/` and visual diffs are archived from `visual_results/`.

## Environment

- Python 3.14 in `.venv/` (arm64 native on M1)
- `setup_mac_mini.sh` for one-time environment setup
- Ollama is optional; `--skip-ollama` uses heuristic scoring instead
